# To Kanban

把"要做什么"的描述，拆成一张符合规则的 Obsidian kanban-plugin 看板。

输入是一份"要做什么"的描述（PRD、需求文档、甚至一句口述都行）。输出是一份 Obsidian kanban-plugin markdown，列管时间窗口、标签管功能模块、勾管状态。

这个 skill **只负责拆卡 + 生成 markdown**，不碰 icafe（建单、回填卡号交给 `icafe-official`）。

## 三条硬规则

拆卡时死守这三条，一条都不能让。

<card-rules>

1. **最小单位**：一张卡 = 一条能跑给人看、1～3 天收口的完整链路，必须带一句验收标准"完成，当……"。塞不进这个窗口的是 epic，不是卡——再拆。

2. **何时该拆**：一行里藏了多个**各自能独立演示**的行为，必须拆。"A + B + C"里只要 A、B、C 各自能单独给人看，就不许挤在一张卡里。

3. **FE/BE 配对**：前后端都涉及的卡，拆成 FE/BE 两张配对卡，并标好依赖；只有一端的活（纯后端兜底、纯前端壳），自然成一张，**不凑空卡**。

</card-rules>

## 卡片形态（固定，不许改）

每张卡严格遵守这个结构：

```
- [ ] [功能域] #占位号 标题一句话 #负责人 ← #依赖占位号
    - 完成，当：……
```

字段从左到右：

| 位置 | 字段 | 说明 |
|------|------|------|
| 1 | `- [ ]` | 状态勾，kanban-plugin 标准开头 |
| 2 | `[功能域]` | 功能模块标签，**从需求里提取**，不写死 |
| 3 | `#占位号` | 卡的身份，依赖引用靠它 |
| 4 | 标题 | 一句话讲清做什么 |
| 5 | `#负责人` | 谁领的活 |
| 6 | `← #依赖占位号` | 我等谁，没有依赖就不写 |
| 子条目 | 验收标准 | "完成，当：……"，必填 |

### 占位号怎么编

`功能域-序号` + 配对后缀（仅 FE/BE 配对卡）。每个功能域**独立从 01 开始**编号，互不干扰。

- 单端卡：`创作-01`、`审核-02`
- FE/BE 配对：`审核-01-BE`、`审核-01-FE`（同一个母号 + 后缀，一眼看出是一对）

### 功能域标签怎么来

`[创作]` `[审核]` 这种标签，**根据输入里实际提到的功能模块来定**，不预设固定清单。换个产品就是另一组标签——skill 的工作是读输入、识别模块、给每个模块一个 `[模块名]`。

### 端类型标签

只在 FE/BE 配对拆开的卡上标 `[FE]`/`[BE]`；纯一端的卡**不标**，标了反而啰嗦。

## 列怎么定（里程碑）

**列 = 时间窗口/里程碑。** 从输入读，分三档：

1. **输入给了具体日期**（如"731 首发、831 增强"）→ 直接照用做列名。
2. **输入只给了阶段、没给日期**（如"先登录、再审核、最后资产"）→ 按阶段建列。
3. **输入啥时间信息都没给** → 只建一列 `📅 待排期`，所有卡先进去，并在文档**开头**加一句提醒：

   ```
   > ⚠️ 里程碑信息缺失，所有卡暂列『待排期』，补上时间窗口后再分列。
   ```

**skill 不擅自编日期。** 假的截止日期比"没给"更糟。

## 工作流

### 1. 读输入，识别功能模块

通读需求，列出"这份需求里到底有哪些功能模块"。这些模块就是功能域标签的来源。

### 2. 拆卡（核心）

对需求里的每一件事，按三条硬规则拆：

- 先问：这件事是不是**一条能 1～3 天收口、跑得通的链路**？不是 → 再拆，或把它归为 epic。
- 再问：它一行里是不是藏了**多个能各自演示**的行为？是 → 拆开。
- 再问：它前后端都涉及吗？是 → 拆 FE/BE 配对 + 标依赖；只有一端 → 一张。

### 3. 标依赖

对每张卡，问"做它之前必须先有谁"。有 → 写 `← #依赖占位号`。

### 4. 定列

按上面的三档规则，决定分几列、每列叫什么。

### 5. 生成 markdown（带中文母语检查）

用下面的模板组装整张看板。**输出前逐卡过一遍中文**：

- 站在中文母语者的角度读，**不出机翻腔、翻译体**（如"纵向切片""可证伪"这类从英文直译的词，换成"一条跑得通的链路""能验证真假"）。
- 技术术语、专有名词、缩写确实比中文清楚的，保留英文（如 `API`、`SSE`、`OAuth`），不强翻。
- 标题一句话、验收标准一句话，都说人话、读着顺。

## 看板模板

```
---
kanban-plugin: board
---

## 🎯 [列名1]

- [ ] [功能域] #占位号 标题 #负责人 ← #依赖占位号
    - 完成，当：……

## 🚀 [列名2]

- [ ] [功能域] #占位号 标题 #负责人
    - 完成，当：……

%% kanban:settings
```
{"kanban-plugin":"board","show-checkboxes":true,"list-collapse":[false]}
```
%%
```

`list-collapse` 的数组长度 = 列数，每项 `false`。

## 一个完整的拆卡例子

**输入**（口述）："审核这块要做：提审发如流通知，顶栏加通过/打回按钮（打回要填意见），后端要兜底校验来源防止越权。先这批，831 前交付。"

**拆出来**：

```
## 🚀 831

- [ ] [审核] #审核-01-BE 提审：审核接口+如流通知 #huabinhong
    - 完成，当：调提审接口后审单状态正确变更 + 如流通知发出

- [ ] [审核] #审核-01-FE 提审入口对接接口 #huabinhong ← #审核-01-BE
    - 完成，当：点提审按钮能调通接口、显示成功/失败态

- [ ] [审核] #审核-02-BE 通过/打回：审批接口（打回意见必填） #huabinhong
    - 完成，当：调通过/打回接口后审单状态变更；打回时意见为空则接口拒绝

- [ ] [审核] #审核-02-FE 顶栏通过/打回态 #huabinhong ← #审核-02-BE
    - 完成，当：顶栏通过/打回按钮可点、点出效果；打回弹出意见框、必填校验

- [ ] [审核] #审核-03 来源硬校验兜底 #huabinhong ← #审核-01-BE
    - 完成，当：构造越权请求（伪造非本人/非本空间来源），后端照样拦住
```

注意这个例子里的几个判断：

- `#审核-01-BE` 和 `#审核-01-FE` 是**配对**（同母号 01 + 后缀）——后端接口独立可演示，前端入口接线依赖它。
- `#审核-03 来源硬校验` 是**纯后端**，不凑 FE 空卡；它依赖 `#审核-01-BE`（得先有提审，才有"来源"可校验）。
- "提审"和"通过/打回"是两件事，各自能独立演示 → **不挤一张卡**，拆成 01 和 02 两个母号。

## 卡号回填（不是这个 skill 的活，但要给它留好接口）

skill 生成的看板用**占位号**。后续用户用 `icafe-official` 建单拿号后，回填成：

```
- [ ] [审核] #审核-01-BE 提审：审核接口+如流通知 #huabinhong → [baidu-...-52](https://console.cloud.baidu-int.com/devops/icafe/issue/baidu-ACG-CXD-CXD-AI-52/show)
    - 完成，当：……
```

回填规则：

- **占位号保留**（`#审核-01-BE`），它是"母号 + 配对"的标识，丢了就看不出 FE/BE 是一对了。
- icafe 真号作为 `→ [号](链接)` 补在**行尾**。
- 卡里的 `← #审核-01-BE` 依赖引用**不动**——它指的是占位号本身，回填后依然成立。
- icafe 内部的卡片关联（系统级），交给 `icafe-official` 用 `--rel-issues` 建。

## 不做的事

- 不碰 icafe（建单/查询/回填是 `icafe-official` 的事）。
- 不擅自编里程碑日期。
- 不把卡拆得过碎（一个 PR/微行为当卡），也不留过大（整个功能域一张卡）。
- 不给纯一端的活凑空卡。
